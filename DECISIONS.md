# Ключевые архитектурные решения (DECISIONS)

1. **Ролевая модель в одной таблице:** Вместо установки тяжеловесных пакетов (вроде Spatie Permission) для простого MVP было принято решение добавить строковое поле `role` (`dispatcher`, `master`) в таблицу `users`. Это делает логику авторизации прозрачной и быстрой.
2. **Использование Laravel Breeze:** Для быстрой реализации надежной аутентификации был выбран стартовый пакет Breeze (на Blade). Он закрывает все вопросы безопасности сессий из коробки.
3. **Защита от состояния "гонки" (Race Condition):**
   Вместо пессимистичных блокировок (`lockForUpdate`), которые могут вызывать дедлоки базы данных, применено атомарное обновление: `RepairRequest::where('id', $id)->whereIn('status', ['new', 'assigned'])->update(['status' => 'in_progress'])`. Если запрос затрагивает 0 строк, система понимает, что заявка уже перехвачена, и отдает ошибку.
4. **UX-оптимизация (Auto-submit форм):**
   Вместо добавления лишних кнопок "Сохранить" для изменения статуса или мастера, в панель управления внедрен onchange-триггер для `select`. Это ускоряет работу диспетчера.
5. **Тестирование логики ролей:**
   Был написан Feature-тест, который физически симулирует создание двух мастеров и двух заявок, а затем проверяет, что Мастер 1 не видит данные Мастера 2 при обращении к дашборду.
